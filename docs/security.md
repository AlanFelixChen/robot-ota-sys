# OTA安全设计文档

## 1. 概述

### 1.1 文档目的
定义机器人端OTA升级系统的**完整安全架构**，覆盖固件签名验签、传输加密、密钥管理、硬件信任根及威胁模型，确保升级全链路安全可信。

### 1.2 安全目标
- 固件完整性：确保固件在传输和存储过程中未被篡改
- 固件真实性：确保固件来源可信，防止恶意固件注入
- 传输机密性：防止固件在传输过程中被窃取或中间人攻击
- 设备身份认证：确保只有合法设备才能获取固件
- 防回滚保护：防止攻击者将设备降级至存在漏洞的旧版本

### 1.3 安全原则
- 纵深防御：多层安全机制叠加，单点失效不影响整体安全
- 最小权限：各组件仅拥有完成功能所需的最小权限
- 硬件信任根：以安全芯片为信任锚点，构建完整信任链
- 密钥不可导出：私钥存储在安全芯片内，任何情况下不可读取

## 2. 安全架构总览

### 2.1 信任链架构

```
┌─────────────────────────────────────────────────────┐
│                    硬件信任根                         │
│              ATECC608 安全芯片                       │
│         （存储设备唯一私钥，不可导出）                   │
└────────────────────┬────────────────────────────────┘
                     │ 验证签名
                     ▼
┌─────────────────────────────────────────────────────┐
│                  安全启动（Secure Boot）              │
│   ROM Boot → U-Boot SPL → U-Boot → Kernel           │
│   每一级验证下一级的ECDSA签名                           │
└────────────────────┬────────────────────────────────┘
                     │ 加载
                     ▼
┌─────────────────────────────────────────────────────┐
│                  OTA升级守护进程                      │
│   固件签名验证 + 版本防回滚 + 传输加密                   │
└────────────────────┬────────────────────────────────┘
                     │ 写入
                     ▼
┌─────────────────────────────────────────────────────┐
│                  固件存储分区                         │
│   App A / App B / 配置分区（RPMB密钥存储）             │
└─────────────────────────────────────────────────────┘
```

### 2.2 安全分层

| 层级 | 安全机制 | 防护目标 |
|------|---------|---------|
| 硬件层 | ATECC608安全芯片 | 密钥安全存储，硬件级防篡改 |
| 启动层 | Secure Boot | 防止启动被篡改的固件 |
| 传输层 | TLS 1.3双向认证 | 传输加密，防窃听和中间人 |
| 应用层 | ECDSA签名验签 | 固件完整性校验 |
| 存储层 | RPMB + 加密分区 | 敏感数据安全存储 |
| 策略层 | 版本防回滚 + 撤销列表 | 防止降级攻击 |

## 3. 硬件信任根

### 3.1 ATECC608安全芯片

| 特性 | 说明 |
|------|------|
| 芯片型号 | Microchip ATECC608A |
| 通信接口 | I2C（地址0x60） |
| 密钥存储 | Slot 0-15，支持硬件锁定 |
| 支持算法 | ECDSA P-256、SHA-256、AES-128 |
| 防篡改 | 主动屏蔽层、电压/温度异常检测 |
| 真随机数 | 内部TRNG，用于密钥生成 |

### 3.2 密钥槽分配

| Slot | 用途 | 权限 | 说明 |
|------|------|------|------|
| 0 | 设备主私钥 | 不可读，仅签名 | 设备唯一身份密钥 |
| 1 | 固件签名公钥 | 可读 | 用于验证固件签名 |
| 2 | TLS客户端私钥 | 不可读，仅签名 | TLS双向认证 |
| 3 | TLS CA公钥 | 可读 | 验证服务器证书 |
| 4 | 数据加密密钥 | 不可读，仅加解密 | 敏感数据保护 |
| 5-7 | 预留 | - | 未来扩展 |
| 8-15 | OTP/配置区 | 锁定后只读 | 设备序列号、硬件版本 |

### 3.3 设备唯一身份

```
设备ID = SHA256(ATECC608序列号 + 设备主私钥公钥)
```

- 设备ID在产线烧录时生成并注册至云端
- 云端通过设备ID关联设备证书和授权策略

## 4. 安全启动（Secure Boot）

### 4.1 启动流程

```
┌──────────┐    验证签名     ┌──────────┐
│ ROM Boot │ ──────────────→│ U-Boot   │
│ (不可改)  │                │ SPL      │
└──────────┘                └────┬─────┘
                                 │ 验证签名
                                 ▼
                            ┌──────────┐
                            │ U-Boot   │
                            │ Proper   │
                            └────┬─────┘
                                 │ 验证签名
                                 ▼
                            ┌──────────┐
                            │ Linux    │
                            │ Kernel   │
                            └──────────┘
```

### 4.2 签名验证流程

```
1. ROM Boot（固化在芯片内，不可修改）
   - 加载U-Boot SPL至SRAM
   - 使用ATECC608 Slot 1公钥验证SPL签名
   - 验证通过 → 跳转SPL
   - 验证失败 → 停止启动

2. U-Boot SPL
   - 初始化DDR和必要外设
   - 加载U-Boot Proper
   - 验证U-Boot Proper签名
   - 验证通过 → 跳转U-Boot Proper

3. U-Boot Proper
   - 加载Linux Kernel和DTB
   - 验证Kernel签名
   - 验证通过 → 启动Kernel

4. Linux Kernel
   - 验证Rootfs签名（dm-verity）
   - 挂载文件系统
```

### 4.3 启动失败处理

| 失败场景 | 处理方式 |
|---------|---------|
| SPL签名验证失败 | 停止启动，进入恢复模式 |
| U-Boot签名验证失败 | 尝试启动备用分区U-Boot |
| Kernel签名验证失败 | 尝试启动备用分区Kernel |
| 备用分区也失败 | 进入USB恢复模式，等待刷机 |

## 5. 固件签名与验签

### 5.1 签名流程（云端）

```
输入：固件镜像（firmware.img）
输出：签名文件（firmware.sig）

步骤：
1. 计算固件SHA256哈希：hash = SHA256(firmware.img)
2. 使用云端ECDSA私钥签名：sig = ECDSA_Sign(hash, private_key)
3. 签名格式：DER编码的ECDSA P-256签名（64字节）
4. 将签名文件与固件一同下发
```

### 5.2 验签流程（设备端）

```
输入：固件镜像（firmware.img）、签名文件（firmware.sig）
输出：验证通过/失败

步骤：
1. 计算固件SHA256哈希：hash = SHA256(firmware.img)
2. 从ATECC608 Slot 1读取公钥
3. 使用公钥验证签名：result = ECDSA_Verify(hash, sig, public_key)
4. 验证通过 → 继续升级流程
5. 验证失败 → 中止升级，上报安全事件
```

### 5.3 密钥管理

| 密钥 | 存储位置 | 生命周期 | 轮换策略 |
|------|---------|---------|---------|
| 云端签名私钥 | HSM硬件安全模块 | 1年 | 年度轮换，旧公钥保留验证 |
| 固件签名公钥 | ATECC608 Slot 1 | 随设备终身 | 支持多公钥共存 |
| 设备主私钥 | ATECC608 Slot 0 | 随设备终身 | 不可轮换，不可导出 |

### 5.4 多公钥支持

```
设备端公钥存储结构：
┌──────────────────────────────┐
│ Slot 1: 当前有效公钥（必选）    │
│ Slot 5: 备用公钥1（可选）      │
│ Slot 6: 备用公钥2（可选）      │
└──────────────────────────────┘

验签策略：
1. 优先使用Slot 1公钥验证
2. 验证失败则尝试Slot 5、Slot 6
3. 任一公钥验证通过即视为有效
4. 支持云端平滑轮换签名密钥
```

## 6. 传输安全

### 6.1 TLS双向认证

```
设备端                          云端
  │                              │
  │──── ClientHello ────────────→│
  │     (支持TLS 1.3)            │
  │                              │
  │←─── ServerHello + 证书 ──────│
  │     (服务器ECDSA证书)         │
  │                              │
  │  验证服务器证书                │
  │  (使用ATECC608 Slot 3 CA公钥) │
  │                              │
  │←─── CertificateRequest ──────│
  │                              │
  │──── 设备证书 + 签名 ─────────→ │
  │     (ATECC608 Slot 2签名)     │
  │                              │
  │←─── Finished ────────────────│
  │                              │
  │←════ 加密通信 ═══════════════→│
```

### 6.2 证书体系

| 证书 | 签发者 | 存储位置 | 有效期 |
|------|--------|---------|--------|
| 根CA证书 | 自签名 | ATECC608 Slot 3 | 10年 |
| 服务器证书 | 根CA | 云端服务器 | 1年 |
| 设备证书 | 根CA | 设备文件系统 | 5年 |

### 6.3 MQTT安全

```
MQTT连接安全配置：
- Broker地址：mqtts://ota-broker:8883
- TLS版本：1.3
- 认证方式：X.509客户端证书
- QoS：QoS 1（至少一次送达）
- Keep Alive：60秒

```

### 6.4 HTTPS下载安全

```
HTTPS下载配置：
- 下载地址：https://ota-cdn/firmware/v2.0.0.img
- TLS版本：1.3
- 认证方式：服务器证书验证

- 下载校验：响应头包含Content-SHA256
```

## 7. 存储安全

### 7.1 分区安全属性

| 分区 | 文件系统 | 加密 | 说明 |
|------|---------|------|------|
| App A | UBIFS | dm-verity（完整性） | 运行分区，只读挂载 |
| App B | UBIFS | dm-verity（完整性） | 备用分区，升级时写入 |
| 数据分区 | UBIFS | dm-crypt（加密） | 日志、临时文件 |
| 配置分区 | UBIFS | RPMB（防篡改） | U-Boot环境变量、OTA状态 |

### 7.2 RPMB安全存储

```
RPMB（Replay Protected Memory Block）特性：
- 硬件级防重放攻击
- 写入计数器，防止回滚
- 共享密钥认证，防止未授权访问
- 存储内容：U-Boot环境变量、启动计数、OTA状态

存储数据：
- boot_part：当前启动分区
- boot_retry_count：启动重试计数
- boot_commit：升级确认标记
- rollback_version：最低允许版本号
```

### 7.3 敏感数据保护

| 数据 | 保护方式 | 存储位置 |
|------|---------|---------|
| 设备私钥 | ATECC608硬件保护 | 安全芯片 |
| 设备证书 | 文件系统（公钥不敏感） | 配置分区 |
| MQTT凭证 | 设备证书认证 | 无需存储密码 |
| OTA下载URL | 加密存储 | 配置分区 |
| 升级日志 | 加密存储 | 数据分区 |

## 8. 防回滚保护

### 8.1 版本号机制

```
版本号格式：主版本.次版本.修订号.构建号（如 2.1.3.42）

防回滚规则：
1. 主版本号不能降低
2. 主版本相同，次版本号不能降低
3. 主次版本相同，修订号不能降低
4. 构建号仅用于同版本重建，不参与防回滚判断
```

### 8.2 防回滚实现

```
设备端防回滚：
1. RPMB中存储 rollback_version（当前最低允许版本）
2. 升级时校验新固件版本 ≥ rollback_version
3. 升级成功后更新 rollback_version = 新版本号
4. 拒绝安装低于 rollback_version 的固件

云端防回滚：
1. 云端记录每台设备的当前版本
2. 下发升级指令时校验目标版本 > 当前版本
3. 特殊回滚需求需加入白名单审批
```

### 8.3 版本撤销

```
场景：发现某个版本存在严重安全漏洞，需紧急撤销

撤销机制：
1. 云端将问题版本加入撤销列表
2. 设备升级前查询撤销列表
3. 目标版本在撤销列表中 → 拒绝升级
4. 设备当前版本在撤销列表中 → 强制升级至安全版本
```

## 9. 威胁模型与防护

### 9.1 威胁识别

| 威胁 | 攻击向量 | 风险等级 | 防护措施 |
|------|---------|---------|---------|
| 固件篡改 | 中间人篡改下载固件 | 高 | ECDSA签名校验 |
| 恶意固件注入 | 伪造升级服务器 | 高 | TLS双向认证 |
| 固件窃取 | 抓包获取固件 | 中 | TLS加密传输 |
| 降级攻击 | 安装旧版本利用已知漏洞 | 高 | 版本防回滚 |
| 重放攻击 | 重放旧升级指令 | 中 | 时间戳+Nonce |
| 密钥提取 | 物理拆解读取密钥 | 高 | ATECC608硬件保护 |
| 侧信道攻击 | 功耗/电磁分析 | 中 | 安全芯片防护 |
| 拒绝服务 | 频繁下发升级指令 | 低 | 升级频率限制 |

### 9.2 攻击场景分析

场景1：中间人篡改固件
```
攻击者截获HTTPS下载流量 → 替换固件内容 → 设备下载被篡改固件
防护：ECDSA签名校验 → 签名不匹配 → 拒绝升级 → 上报安全事件
```

场景2：伪造升级服务器
```
攻击者搭建伪造OTA服务器 → 下发恶意升级指令 → 设备连接伪造服务器
防护：TLS双向认证 → 服务器证书验证失败 → 连接中断
```

场景3：物理攻击提取密钥
```
攻击者拆解设备 → 尝试读取Flash中的密钥 → 获取私钥
防护：私钥存储在ATECC608 → 芯片防篡改 → 物理攻击触发密钥自毁
```

场景4：降级攻击
```
攻击者获取旧版本固件 → 伪造升级指令指向旧版本 → 设备安装有漏洞旧版本
防护：RPMB防回滚版本号 → 版本低于最低允许版本 → 拒绝升级
```

## 10. 安全事件响应

### 10.1 安全事件等级

| 等级 | 事件 | 响应时间 | 处理方式 |
|------|------|---------|---------|
| 严重 | 签名验证失败、密钥泄露 | 立即 | 暂停所有升级，安全团队介入 |
| 高危 | 证书过期、撤销列表命中 | 1小时 | 暂停相关设备升级 |
| 中危 | 下载异常、校验失败 | 24小时 | 记录日志，自动重试 |
| 低危 | 升级频率异常 | 7天 | 监控告警 |

### 10.2 设备端安全上报

```
上报事件类型：
- SEC_SIGNATURE_FAIL：固件签名验证失败
- SEC_CERT_EXPIRED：证书即将过期（提前30天）
- SEC_ROLLBACK_BLOCKED：防回滚拦截
- SEC_REVOKED_VERSION：版本在撤销列表中
- SEC_TLS_ERROR：TLS握手失败
- SEC_TAMPER_DETECTED：ATECC608篡改检测

上报格式：
{
  "event": "SEC_SIGNATURE_FAIL",
  "device_id": "xxx",
  "timestamp": 1234567890,
  "detail": {
    "firmware_version": "2.0.0",
    "expected_hash": "a1b2c3...",
    "actual_hash": "d4e5f6..."
  }
}
```
