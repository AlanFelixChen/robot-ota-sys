# OTA差分升级设计文档

## 1. 概述

### 1.1 文档目的
定义机器人端**OTA差分升级**的完整技术方案，包括差分算法选型、差分包生成流程、设备端还原流程及性能指标，为开发实现和测试验证提供依据。

### 1.2 差分升级原理
差分升级（Delta Upgrade）仅传输新旧版本固件之间的差异数据，而非完整固件包。设备端下载差分包后，结合本地旧版本固件还原出完整的新版本固件，从而大幅减少下载数据量。

### 1.3 适用场景
- 固件体积较大（>500MB），全量下载耗时长
- 设备网络带宽受限（WiFi/4G弱信号环境）
- 版本间差异较小（<30%），差分收益显著
- 批量设备同时升级，节省云端带宽成本

## 2. 差分算法选型

### 2.1 候选方案对比

| 方案 | 压缩率 | 内存占用 | 还原速度 | 流式支持 | 成熟度 |
|------|--------|---------|---------|---------|--------|
| bsdiff | 极高（5%-15%） | 高（需加载完整旧固件） | 慢 | 不支持 | 成熟 |
| xdelta3 | 中等（20%-40%） | 低 | 快 | 支持 | 成熟 |
| zchunk | 高（10%-30%） | 低（分块处理） | 快 | 原生支持 | 较新 |
| casync | 高（10%-25%） | 低 | 快 | 支持 | 较新 |

### 2.2 选型结论：zchunk

选择理由：
- 压缩率高：实测固件差分包大小为全量包的10%-30%
- 内存友好：分块处理，单块内存占用<10MB，适合嵌入式设备
- 流式还原：支持边下载边还原，无需完整下载后再处理
- 断点续传：分块独立校验，下载中断可从断块恢复
- 生态兼容：与SWUpdate原生集成，减少开发工作量

### 2.3 zchunk工作原理

```
原始固件（新版本）
        │
        ▼
┌──────────────┐
│  分块切割     |  按固定大小（如4MB）切割固件
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  块哈希计算   │  计算每块的SHA256哈希
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  差异检测     │  与旧版本对应块比较，标记相同/差异块
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  差异压缩     │  对差异块进行zstd压缩
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  元数据打包   │  块索引 + 差异数据 + 校验信息
└──────┬───────┘
       │
       ▼
  差分包（.zck）
```

## 3. 差分包生成流程（云端）

### 3.1 生成流程

```
输入：旧版本固件（v1.0.0.img）、新版本固件（v2.0.0.img）
输出：差分包（v1.0.0_to_v2.0.0.zck）

步骤：
1. 加载旧版本固件，建立块哈希索引
2. 加载新版本固件，按块大小切割
3. 逐块匹配：
   - 哈希相同 → 标记为"引用块"，记录旧固件中的偏移
   - 哈希不同 → 标记为"差异块"，计算差异数据并zstd压缩
4. 生成块索引表（块序号、类型、偏移、大小、校验和）
5. 打包为.zck文件
6. 使用ECDSA私钥对差分包签名
```

### 3.2 块大小选择

| 块大小 | 压缩率 | 还原速度 | 内存占用 | 推荐场景 |
|--------|--------|---------|---------|---------|
| 1MB | 最优 | 较慢 | 低 | 差异极小 |
| 4MB | 良好 | 适中 | 适中 | 推荐默认值 |
| 8MB | 一般 | 快 | 较高 | 差异较大 |

### 3.3 多级差分策略

对于跨版本升级（如v1.0.0 → v3.0.0），支持两种策略：

| 策略 | 方式 | 优点 | 缺点 |
|------|------|------|------|
| 直接差分 | 生成v1.0.0→v3.0.0差分包 | 一次下载 | 差分包可能较大 |
| 多级差分 | 依次下载v1→v2、v2→v3差分包 | 单包更小 | 需多次还原 |

默认策略：优先使用直接差分，当直接差分包>全量包50%时，自动切换为多级差分。

## 4. 设备端还原流程

### 4.1 还原流程

```
输入：差分包（v1.0.0_to_v2.0.0.zck）、当前固件分区（App A，v1.0.0）
输出：完整新固件（写入App B分区）

步骤：
1. 下载差分包至临时存储区
2. 验证差分包ECDSA签名
3. 解析差分包块索引表
4. 逐块还原：
   ┌─ 引用块 → 从当前固件分区读取对应块，直接写入目标分区
   └─ 差异块 → 从当前固件分区读取对应块 + 差异数据 → 合并 → 写入目标分区
5. 校验还原后固件的SHA256（与云端下发哈希比对）
6. 清理临时文件
```

### 4.2 内存优化设计

```
┌─────────────────────────────────────┐
│            内存布局（<50MB）          │
├──────────┬──────────┬───────────────┤
│ 块索引表  │ 当前块数据 │ 差异数据缓冲区  │
│  ~2MB    │  ~4MB    │    ~8MB       │
└──────────┴──────────┴───────────────┘

- 块索引表：一次性加载，常驻内存
- 当前块数据：按需从Flash读取，单块处理完释放
- 差异数据缓冲区：解压差异数据，合并后写入Flash
```

### 4.3 断点续传支持

```
差分包下载中断恢复：
  - 差分包按块独立校验
  - 记录已下载完成的块序号
  - 恢复下载时从下一个未完成块开始
  - 已下载的块数据保留在临时存储

还原中断恢复：
  - 记录已还原完成的块序号
  - 恢复时从下一个未还原块继续
  - 目标分区已写入的块数据有效
```

## 5. 性能指标

### 5.1 压缩率实测数据

| 固件类型 | 全量大小 | 版本差异 | 差分包大小 | 压缩率 |
|---------|---------|---------|-----------|--------|
| Linux内核 | 120MB | 小版本升级 | 8MB | 6.7% |
| Rootfs | 800MB | 功能更新 | 120MB | 15% |
| AI模型 | 500MB | 模型迭代 | 80MB | 16% |
| 完整固件 | 1.5GB | 季度更新 | 250MB | 16.7% |

### 5.2 还原耗时

| 固件大小 | 差分包大小 | 还原耗时 | 全量写入耗时 | 节省时间 |
|---------|-----------|---------|-------------|---------|
| 1.5GB | 250MB | 3min | 8min | 62.5% |
| 800MB | 120MB | 1.5min | 4min | 62.5% |
| 500MB | 80MB | 1min | 2.5min | 60% |

### 5.3 资源消耗

| 指标 | 差分还原 | 全量写入 | 说明 |
|------|---------|---------|------|
| 峰值内存 | <50MB | <20MB | 差分需额外块索引和差异缓冲 |
| CPU占用 | 30%-50% | 10%-20% | 差分需解压和合并计算 |
| Flash写入量 | 等于固件大小 | 等于固件大小 | 最终写入量相同 |
| 下载流量 | 10%-30% | 100% | 差分显著节省 |

## 6. 异常处理

### 6.1 异常场景与处理

| 异常场景 | 检测方式 | 处理策略 |
|---------|---------|---------|
| 差分包损坏 | 块校验失败 | 重新下载损坏块 |
| 差分包签名无效 | ECDSA验签失败 | 中止升级，上报安全事件 |
| 当前固件不匹配 | 块哈希与索引不匹配 | 中止差分，回退全量升级 |
| 还原后校验失败 | SHA256比对不匹配 | 重新还原，仍失败则回退全量 |
| 存储空间不足 | 写入前检查 | 中止升级，提示清理空间 |
| 还原过程中断电 | 重启后检测未完成 | 从断点继续还原 |

### 6.2 降级策略

当差分升级失败时，自动降级为全量升级：

```
差分升级失败
    │
    ├── 差分包损坏 → 重新下载差分包（1次）
    │       │
    │       └── 仍失败 → 降级全量升级
    │
    ├── 还原失败 → 重新还原（1次）
    │       │
    │       └── 仍失败 → 降级全量升级
    │
    └── 当前固件不匹配 → 直接降级全量升级
```

## 7. 与SWUpdate集成

### 7.1 SWUpdate配置

```
swupdate.cfg 差分升级相关配置
suricatta {
    enable = true;
    url = "https://ota-server/updates";
    delta = true;              启用差分升级
    delta-max-size = 30;       差分包最大占比30%，超过则全量
}

download {
    retries = 3;
    timeout = 600;
    resume = true;             启用断点续传
}

delta {
    engine = "zchunk";         差分引擎
    block-size = 4096;         块大小4MB（单位KB）
    temp-path = "/data/ota/tmp";
    verify-after-restore = true;
}
```

### 7.2 差分包描述文件

```yaml
sw-description
software:
  version: "2.0.0"
  hardware-compatibility: ["rev-b", "rev-c"]
  
  images:
    - filename: "rootfs_v2.0.0.img"
      type: "delta"
      device: "/dev/mmcblk0p3"
      delta-from: "1.0.0"
      sha256: "a1b2c3d4..."
      
      files:
        - filename: "rootfs_v1.0.0_to_v2.0.0.zck"
          sha256: "e5f6g7h8..."
```
